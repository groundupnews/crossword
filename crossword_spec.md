# Architecture

## Backend

- Django/Python version 6
- Sqlite database
- Linux
- In development we will be using the Django ./manage.py server
- In production we will use Nginx and Gunicorn on an Ubuntu server

## Frontend

- HTML first with SVG elements
- Only vanilla Javascript, and kept to a minimum. No frameworks except HTMX.
- HTMX to communicate with the server
- Plain CSS. No Tailwind or other 3rd-party add-ons.

# Database Design

Note on terminology: a record in `Word` may contain more than one natural-language
word (e.g. a multi-word phrase), but is stored as a single A-Z string.

Model: Word
- pk
- text: (must be string of A-Z, stored uppercase; validated at the model
  level via a validator (hard validation: raises on save, no silent
  coercion) and at the form/UX level)
- date_added
- date_modified
(text is unique)

Model: Clue
- pk
- text: foreignkey to Word
- clue: (unicode)
- date_added
- date_modified
(text and clue are unique together)

Model: Crossword
- pk
- name (optional)
- description (optional)
- date_added
- date_modified
- publication_datetime (default to null, can be in the future)
- num_rows
- num_cols
- blocked_out_squares (JSONField, list of integers)
- cells (JSONField, row-major list of letters and blanks, same length as
  the grid; this is the source of truth for the grid contents)
- requires_rotational_symmetry (boolean, default true)

Model: Entry
- pk
- crossword foreignkey
- word foreignkey
- clue foreignkey (can be null)
- number (integer: the grid number the entry starts at)
- direction (across or down)
(crossword, number, and direction are unique together)

Note on the data model: `cells` on `Crossword` is the source of truth for the
grid. `Entry` records are a derived, queryable index over the completed slots,
regenerated on each save. A partial (incompletely filled) slot is preserved in
`cells` but produces no `Entry` record. Start row/col for an entry are not
stored; they are calculated from the grid geometry and the entry number.

# User interface

## Adding a new crossword

User enters a form with the following fields:
- Grid size - choice of 15x15 or 19x19
- Requires rotational symmetry - default to true

Upon entering the above, a blank crossword is saved in the database.

Then this crossword is automatically selected and the user is redirected to the crossword edit screen.

This is implemented with a plain Django ModelForm (two fields: grid size, NYT
rules) and a CreateView that redirects to the edit screen. The Django admin is
not used for this flow, but remains enabled separately as a data-inspection/
debugging tool.

## Select crossword

- Standard form to select a saved crossword.

Implemented as a plain Django view/form (not the admin), for consistency with
the add and edit screens.

## Editing a crossword

User is shown an editable crossword grid in one pane.
Above the grid are editable fields to change the name and description of the crossword.
The date added and modified fields can be displayed discreetly.

There is a save button underneath the grid.

At some point, not now, we'll build an undo feature.

The user can enter answers in the cells in the grid.
The user can also block cells with the spacebar. If rotational symmetry is required, symmetry is maintained. If the spacebar is entered on a cell with a letter, it's erased and replaced with a black block, no questions asked (same for symmetric block that may have to be created).
Each time a cell is updated, the grid across and down numbers are updated. If necessary, a clue is deleted.
The user can enter clues for each answer.

If the user enters an answer whose text already appears elsewhere in the same
crossword, a non-blocking warning is shown. The user may proceed.

If the user clicks a button saying "Fetch answers", up to 20 matching possible
Word entries (text field in the Word model) for the current across or down
answer are fetched and displayed in a pane next to the crossword. The user can
select one of these. If there are more possible entries than 20 they should be
randomly selected, but displayed in alphabetical order. This is just to assist
the user. If the user wants to enter text freely, creating entries not in the
Word table, that's fine.

If the user clicks a button saying "Fetch clues", an analogous process for
clues takes place (up to 10 matching clues, randomly selecting if more
than 10 possible).

## Saving

When the user presses save, the full grid contents are written to the `cells`
field of the Crossword (this is the source of truth and always saved, even
mid-word). The save then derives the completed slots from `cells`:

- For each slot that is completely filled, the answer text is written to the
  Word table (if not already present) and a corresponding Entry row is created
  or updated, linking the crossword, word, number, direction and (if entered)
  clue. Clues are written to the Clue table.
- A slot that is only partially filled is preserved in `cells` but produces no
  Word, Clue or Entry record.
- The Crossword and Entry tables are updated appropriately.

## JSON export

There is a JSON button which generates a JSON representation of the crossword,
read from the `cells` source of truth and the associated Entry/Clue records.

# grid.py — slot detection and numbering

A pure-logic module, no Django dependency. Its job is to turn the stored grid
state into the numbered across/down slots that the save view consumes and that
the edit-screen JavaScript mirrors for live renumbering. Defining it once here
keeps client and server in agreement.

## Conventions

- Cell indexing is row-major single integers: index = row * num_cols + col.
- `blocked_out_squares` is the sole authority on which cells are black. `cells`
  holds letters and is only meaningful for white (non-black) cells.
- A slot is a maximal run of two or more contiguous white cells in a row
  (across) or column (down). Isolated single white cells are not slots and are
  not numbered.
- Numbering follows standard crossword convention: scan cells in row-major
  order; a cell receives the next number if it begins an across slot, a down
  slot, or both. A cell begins an across slot if it is white, has no white cell
  to its left (edge or black), and has a white cell to its right. Likewise for
  down (no white cell above, white cell below). The same number serves both the
  across and down slot that start at that cell.
- A slot is "complete" when every one of its cells holds a letter. Completeness
  has no other meaning; the save view skips incomplete slots.

## Interface

Given num_rows, num_cols, blocked_out_squares and cells, the module reports the
list of slots, each carrying: number, direction (across/down), start index,
length, the list of cell indices it covers, its current letters, and whether it
is complete.

# Save POST payload

The client holds the authoritative grid state and sends it as a single JSON
body:

- `cells`: row-major list of letters and blanks (the source of truth, sent
  verbatim).
- `blocked_out_squares`: list of black-cell indices.
- `name`, `description`: the two editable text fields.
- `clues`: mapping from slot identity (number + direction, e.g. "1A") to clue
  text. Only slots the user wrote a clue for need appear. The client keys these
  to the current numbering at save time (it renumbers live for display anyway),
  so the server trusts the payload's keys — a clue follows the word because the
  client re-keys it on every renumber.

The save view then:
- Writes cells, blocked_out_squares, name and description to the Crossword.
- Runs grid.slots().
- For each complete slot: upserts the Word, upserts any Clue keyed by
  (word, clue text), upserts the Entry keyed by (crossword, number, direction).
- Deletes Entry rows whose (number, direction) no longer corresponds to a
  complete slot.
- Partial slots touch nothing but cells.

# JSON export shape

Read from cells and the Entry/Clue records. Self-contained:

    {
      "name": "...",
      "description": "...",
      "size": {"rows": 15, "cols": 15},
      "blocked_out_squares": [3, 11, ...],
      "grid": ["C", "A", "T", ...],
      "entries": [
        {"number": 1, "direction": "A", "answer": "CAT", "clue": "Feline"},
        ...
      ]
    }

- `grid` holds letters, with empty string for blanks; blocks are marked only via
  blocked_out_squares (no redundant encoding).
- `entries` come only from complete slots (incomplete slots have no Entry).

# Fetch answers / fetch clues

Both endpoints act on the current slot, identified by the live client state
(the client sends the slot pattern/word directly, not a cell reference, so the
match uses what the user currently sees rather than possibly-stale saved cells).

Fetch answers:
- The client sends the slot pattern: filled positions as their letter, blank
  positions as "?" (e.g. "C?T").
- The server matches Word.text using SQLite GLOB against that pattern, which
  enforces both exact length and fixed letters ("?" is a single-character
  wildcard).
- If more than 20 match, randomly select 20, then sort alphabetically for
  display. Returns {"answers": [...]}.

Fetch clues:
- The client sends the complete current word. If the slot is not complete there
  is no single word to match and the result is empty.
- The server returns the clue text of all Clue rows whose word is that word.
- If more than 10 match, randomly select 10, then sort for display. Returns
  {"clues": [...]}.

Selecting a result from the side pane fills the slot's cells (answers) or sets
the clue input (clues), leaves the cursor at the slot start, and re-renders.
